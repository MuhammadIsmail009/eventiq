# DESIGN: EventIQ dashboard

Written 2026-07-14. The dashboard is a single self-contained HTML file
generated by the Python tool. Dark theme first. Every colour value and
contrast ratio below was computed and checked against WCAG AA (sources in
RESEARCH.md section B).

---

## 1. Direction: "Terminal Editorial"

One paragraph. A dark, high-signal SOC dashboard that borrows typographic
confidence from the studio references without their whitespace. From
**kargo-studio** we take the core thesis: hierarchy comes from type *weight*,
not colour, so a dense grid can still feel composed. From **higgsfield.ai** we
take the surface architecture: near-black charcoal panels built as an elevation
ramp with alpha-white borders, grotesk headers over monospace data. From
**wairk.fr** we take charcoal-not-black grounds and the discipline of exactly
one accent. The editorial move (oversized, confident type) is spent once per
view on the primary KPI number; everywhere else the type shrinks to 11-13px and
the weight contrast does the work. Monospace carries every IP, port, hash,
timestamp and command, because column alignment and 0-versus-O legibility is the
whole job of a log tool.

Tension resolved: portfolio sites earn presence through scale and empty space,
which a triage grid cannot afford. We transplant the *confidence* (weight and
case discipline) and drop the *emptiness*. Whitespace is spent at the top; the
grid gets to work below.

## 2. Colour tokens (dark theme; ratios computed vs stated background)

Surfaces, an elevation ramp (near-black base, not pure `#000`, so text avoids
halation and there is room to elevate):

```
--bg-base        #0E1116   app background
--surface-01     #161B22   panels, cards, table body
--surface-02     #1C232D   row hover, raised
--surface-03     #242C38   popover, flyout
--border-subtle  #2A313C   grid lines
--border-strong  #3A424E   active / selected border
```

Text tiers (contrast vs `#0E1116` / vs panel `#161B22`):

```
--text-primary    #E8ECF1   15.9 / 14.6   values, headers
--text-secondary  #98A2B0    7.3 /  6.7   labels
--text-tertiary   #818C9B    5.6 /  5.1   metadata (clears AA small text)
--text-data(mono) #C6CDD6   11.8 / 10.8   IPs, hashes, commands
```

Accent, exactly one, non-neon: `--accent #78A9FF` (8.0:1) for focus rings,
selection, links. No lime, no glow, no saturated fill.

Severity, four steps, colour AND shape AND label (colour alone fails ~8% of male
analysts; see RESEARCH.md B). Ratios vs base:

```
CRITICAL  #FF6F4B  6.9:1   filled solid chip, dark ink label, glyph ●
HIGH      #E69F00  8.4:1   outlined chip,                    glyph ◆
MEDIUM    #F0E442 14.3:1   tinted underline bar,             glyph ▲
LOW       #56B4E9  8.2:1   text + dot only,                  glyph ■
INFO      #818C9B  5.6:1   plain grey text
RESOLVED  #42BE65  7.9:1   check + strikethrough
```

The glyph column is mandatory, not decorative: it makes severity legible in pure
grayscale and under deuteranopia, where CRITICAL and HIGH sit only 1.1:1 apart in
colour. Every real SOC (Splunk, Defender, Wazuh) encodes severity redundantly;
so do we.

## 3. Typography

```
Display / headers   Space Grotesk    weights 500 / 600 / 700
Body / UI           Inter            weights 400 / 500 / 600
Data / mono         IBM Plex Mono    IPs, ports, hashes, rule.id, commands, time
```

Scale, dense, on a 4px rhythm:
`11 (mono caps label, +0.04em) · 12 · 13 (table/body) · 14 · 16 · 20 · 28 ·
48 (the one editorial KPI number)`.

Rules: uppercase + tracking only on <=11px mono labels (column heads, sev tags),
never on data values. Weight contrast within a row (400 data against 600
emphasis) is the Kargo lesson applied at small size.

Fonts are embedded in the single HTML file as base64 `@font-face` (WOFF2) so the
file has zero external requests and renders under a strict CSP. If embedding
pushes the file size too high for comfort, fall back to a system stack
(`ui-monospace, "Cascadia Mono", Consolas` for data; `system-ui` for text) and
note the tradeoff. Decision deferred to build; correctness first, embedded fonts
second.

"export" is wired; the rest demonstrate the shape). Sort and severity/rule
filters are client-side JS over the embedded JSON.

## 5. Component inventory (with states)

| Component | States |
|---|---|
| KPI stat tile | default; the one 48px hero number; optional Δ-vs-prior arrow |
| Severity chip | critical (filled) / high (outline) / medium (underline) / low (dot) / info / resolved; each with its glyph; hover shows the Wazuh level |
| Alert table row | default; hover (`surface-02`); selected (accent left-border); expanded (flyout open) |
| Bulk action bar | hidden; visible when >=1 row selected; export enabled, others disabled-with-tooltip in v1 |
| Time-series area | severity-stacked; hover tooltip with per-bucket counts |
| Horizontal bar (top rules / entities) | bar-in-cell; hover; click navigates to filtered alert view |
| MITRE lane | tactic nodes left to right; a node is filled if covered, hollow if not |
| Filter controls | dropdowns for severity / rule / group-by; text search; all client-side |

## 6. Data-to-pixel mapping (which field feeds which pixel)

| Component | Source field(s) in the JSON report |
|---|---|
| KPI: critical / high counts | count of alerts by `rule.level` band |
| KPI: distinct attacks | distinct `rule.id` with >=1 alert |
| KPI: entities at risk | count of entities with `risk_score` > 0 |
| Alerts-over-time | alert `first_seen` bucketed, stacked by severity |
| Top rules | `rule.id` -> count |
| MITRE coverage | union of `rule.mitre.id` across alerts, grouped by tactic |
| Top entities | `entity_risk[].{entity, risk_score, severity}` |
| Alert row | `rule.id`, `rule.level`->chip, `first_seen`, `data.{srcip,dstip,...}`, `status` |
| Flyout "why" / evidence | `title`, `evidence{...}`, `mitre.id`, `rule.level`, `sample_rows[]` |
| Incident view | `incidents[].{primary_entity, kill_chain, narrative, alerts[]}` |

Severity chip maps from Wazuh level: 15 and 14 -> CRITICAL, 12-13 -> HIGH,
10-11 -> MEDIUM, below 10 -> LOW/INFO. That keeps the dashboard speaking the
Wazuh level the analyst already reads.

## 7. What we are NOT doing (non-goals)

- No decorative motion. Motion budget is <=150ms state feedback only (hover,
  focus, row expand). No scroll-jacking, no count-up on numbers that must be read
  now, no entrance animations on table rows.
- No pie, donut, gauge, radial "cyber" ring, or 3D chart. Stacked area and
  horizontal bar only.
- No neon-on-black, no glow/box-shadow bloom, no pure `#000`.
- No colour-only severity. Ever.
- No external requests: fonts, styles, scripts and data all inline, so the file
  works offline and under a strict CSP.
- No generic framework look (default-blue primary, light cards, shadow on
  everything). The mono data columns, weight-based hierarchy and single accent
  are the anti-generic markers.
